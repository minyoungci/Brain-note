### 1. 체크포인트 선택 기준 불일치 — 방법론 버그 (Archive/ACCV/EXPERIMENT_LOG.md, Stage 1 vs Stage 2 섹션)

Stage 1 단일 시드 결과(oracle 0.859)는 **pooled AUC 기준으로 체크포인트 선택**했고, Stage 2 멀티시드 재실험은 **macro AUC 기준**으로 전환하였다. 이 전환이 두 스테이지 비교를 구조적으로 불공정하게 만든다. 같은 seed-10에서 oracle이 0.859 → 0.823으로 떨어진 이유가 바로 이것이며, 연구팀 스스로 이를 "lucky volatile pooled-val checkpoint artifact"로 진단하였다. Stage 2 섹션은 이 원인을 명시적으로 기술하고 있으므로, "보고 내에서 충분히 명시되지 않았다"는 초안의 평가는 다소 과장이다. 실제 문제는 Stage 2가 기준을 수정하면서 두 스테이지 간 수치를 직접 비교할 수 없음에도, 결과 표를 나란히 제시한 점이다.

## 코드 리뷰 (diff 기반)

### 버그

**B1. `fidelity_eval` 내 generate 출력 디코딩 — 메커니즘 서술 부정확 (train_vlm.py)**
초안은 `regen_medgemma.py`의 `out[:, enc["input_ids"].shape[1]:]` 슬라이싱과 비교하나, 두 호출은 성격이 다르다. `regen_medgemma.py`는 `input_ids`를 넘기므로 `out`에 입력 IDs가 앞에 포함된다. `fidelity_eval`은 `inputs_embeds`만 넘기고 `input_ids`는 없으므로, HF 구현에 따라 `gen`이 순수 생성 토큰만 담을 수 있다. 실제로 K개 garbage 토큰이 붙는지 여부는 HF 버전·모델별로 다르다. [불확실] — 현재 환경에서 `gen.shape[1]`이 `K + max_new_tokens`인지 `max_new_tokens`만인지 실행 확인 필요.

**B2. smoke_forward.py에 BOS 마스킹 누락 (smoke_forward.py vs train_vlm.py)**
`train_vlm.py`는 `lab[:, K] = -100`으로 visual 직후 BOS를 예측 타깃에서 제외하나, `smoke_forward.py`의 `lab` 구성에는 이 처리가 없다. smoke가 실제 학습 forward를 충실히 재현하지 못한다. ✓ (diff로 확인)

**B3. EOS 이중 추가 가능성 (train_vlm.py:~65)** [불확실]
`ids = [tok(...).input_ids + [tok.eos_token_id] for m in mgL]` — MedGemma 토크나이저가 이미 EOS를 추가하는지 diff에서 확인 불가. 확인 필요.

**B4. `mode_of(t)`가 `df`(OUTF)의 `findings` 키 참조 (regen_medgemma.py)** [불확실]
`df`는 `medgemma_27b_reports.jsonl`에서 읽는다. OUTF의 레코드가 `findings` 키를 포함하는지 diff에서 확인 불가 — 병합 단계(`report_llm`, `fidelity_pass`, `fidelity_errors`만 교체)로는 `findings` 존재를 보장하지 않는다. KeyError 위험.

**B5. `fidelity_eval` 생성 시 BOS 토큰 누락 (train_vlm.py:~155) — 초안 누락**
학습 시 입력은 `[vis_K | BOS | report_text]` (MedGemma 토크나이저는 BOS를 자동 삽입)이며, `lab[:, K] = -100`으로 BOS를 예측 타깃에서만 제외한다. 그러나 `fidelity_eval`의 `generate(inputs_embeds=vis)`는 visual K개 토큰만 context로 제공하고 BOS가 없다. 학습과 추론의 prefix 구조가 다르므로 생성 품질이 저하될 수 있다.

### 데이터 누수

**L1. 분할 상속 위험 (train_vlm.py)**
`sp = meta["split"]`이 G0 npz를 그대로 재사용한다. G0 하이퍼파라미터가 이 val 분할로 튜닝됐다면 VLM val/test 독립성이 약화된다. 단, G0 튜닝 이력은 이번 diff에 나타나지 않아 전제가 추정이다. [불확실] — 분할 독립성 명시 여부 확인 권장.

**L2. 재생성 시 test-set fidelity_gate 노출 (regen_medgemma.py)**
`fail_tags = [t for t, r in df.items() if not r.get("fidelity_pass", True)]`는 split 정보를 참조하지 않는다. `G.check(recs[j], rg[k])`에서 test 분할의 findings가 재생성 루프에 노출된다. ✓ (diff로 확인)

### 테스트 누락

**T1. collate right-padding vs. 추론 left-padding 미검증 (train_vlm.py)**
`collate`의 `ti[j, :len(x[1])] = x[1]`은 right-padding이다. `regen_medgemma.py`는 `tok.padding_side = "left"`를 명시하는데, 이는 배치 생성(추론) 컨텍스트이므로 학습 collate와 용도가 다르다. 학습 시 right-padding은 일반적이나, 이 차이가 의도적임을 확인하는 단위 테스트가 없다. ✓ (diff로 확인)

**T2. aux head grad 미검증 (smoke_forward.py)**
`ok` 조건이 `g_proj AND g_enc AND g_lm is None`이나, `aux.weight.grad`가 흐르는지 smoke에서 확인하지 않는다. ✓ (diff로 확인)

### 재현성

**R1. DataLoader worker seed 미설정 (train_vlm.py)**
`num_workers=6`에 `worker_init_fn`이 없어 worker 내부 random state가 비결정적이다. `torch.Generator`가 shuffle 순서를 고정하나, worker 내 augmentation이 추가될 경우 재현성이 깨진다. ✓ (diff로 확인)
