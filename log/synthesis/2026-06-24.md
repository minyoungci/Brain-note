## 주목·결정 필요 (우선순위순)

**1. `use_skip` 비대칭 — headline 결과를 흔들 수 있는 confound (★당신이 놓쳤을 가능성)**
일일 분석은 "s3d vs skip-free 기준 Δ수렴"을 핵심 진전으로 보고했으나, `resenc_s3d`는 `use_skip=True`, 비교 기준인 `resenc_mae`는 skip-free다. **skip connection은 seg에서 단독으로 이득을 주는 것이 정설**이므로, Δ−0.046→−0.006 개선분이 S3D masked-conv pretraining이 아니라 **encoder의 skip 구조 차이**에서 왔을 가능성을 배제하지 못한다. 일일 분석은 구조 차이를 "직접 비교 불가"로만 적었지 이를 핵심 수치의 교란요인으로 연결하지 않았다 — 같은 encoder 구조(skip on/off 통제)에서 S3D vs 일반 MAE를 재비교하지 않는 한 이 진전 주장은 약하다. **결정:** s3d 학습 효과와 skip 효과를 분리하는 통제 실험 필요.

**2. train.py S3D 배선 + 누수 검증 코드 부재 — 수치 신뢰의 게이트**
step40k 수치(masked-voxel MSE 수렴)는 train.py가 실제로 `forward_masked_backbone`을 호출하고 loss가 hidden 위치만의 MSE라는 전제에 **전적으로 의존**한다. 그런데 train.py +9/-1과 누수 검증("+100 변경 시 recon 차이 0.000e+00")의 재현 코드가 어떤 커밋에도 없다. 배선이 full-volume MSE로 잘못 됐다면 step40k 수치의 의미가 달라진다 — 즉 1번과 합쳐 **현재 headline 결과는 미검증 배선 위에 서 있다.** [근거부족] **결정:** train.py diff 확인 + 누수 단위 테스트 커밋.

## 코드 리뷰

### `pretrain/models.py` — `forward_masked` 경계 re-zero 설계와 누수 검증

```python
m = vis
x = self.stem(x * vis) * m   # 코드 주석: "입력 마스킹 + stem 후 re-zero"
```

`m = vis` 초기화 후 입력 pre-zeroing(`x * vis`)과 출력 re-zeroing(`* m`)이 이중으로 적용된다. 코드 주석 상 의도적 설계이며(conv 수용장을 통한 경계 정보 유입 차단), SCRATCHPAD에 "masked 입력 +100 변경 시 recon 차이 0.000e+00(누수 완벽 0) vs 일반 forward 11.99" 검증 결과가 기술돼 있다. 그러나 이 검증의 **재현 가능한 코드가 어떤 커밋에도 없어** 독립 재검증 불가 [근거부족: 결과는 SCRATCHPAD 기록이나 검증 코드 자체는 diff에 없음].

### `pretrain/models.py` — `use_skip` 공유와 eval 경로 비대칭

```python
# SSLModel.__init__ (diff 확인)
self.backbone = ResEncUNet(use_skip=(cfg["dense"] == "s3d"))
```

`resenc_s3d` recipe는 `use_skip=True`로 인스턴스화된다. `forward(x)` (일반 eval 경로)와 `forward_masked_backbone(x, vis)` (S3D 학습 경로)가 동일한 `self.backbone`을 공유하므로, eval 시 `forward(x)`를 호출하면 skip이 켜진 상태로 **비마스킹** 일반 forward가 실행된다. `resenc_mae`(skip-free)와 다른 encoder 구조이므로 두 recipe의 downstream eval 결과는 구조 차이로 직접 비교 불가할 수 있다. ### `pretrain/eval_harness.py` — spawn Pool 반복 생성 오버헤드

```python
with get_context("spawn").Pool(min(workers, len(args))) as pool:
    res = pool.map(_load_one, args, chunksize=1)
```

`load_task_parallel`은 task 루프(직렬 루프 교체 위치) 안에서 호출되므로 task마다 spawn Pool을 생성·소멸한다. `spawn`은 Python 인터프리터를 완전 재시작하므로 task 수 × spawn 오버헤드가 누적된다. 기능 버그는 아니지만 multi-task eval에서 병목이 될 수 있다. [diff 확인: `load_task_parallel` 내부 with-Pool 구조·직렬 루프 제거 실재] `_load_one`을 top-level 함수로 분리한 것은 `spawn` 피클링 요건에 따른 올바른 설계다.
