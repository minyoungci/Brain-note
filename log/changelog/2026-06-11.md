## 왜

커밋 메시지와 README 개편 내용으로부터 추론한 배경:

1. **수동 갱신의 한계 극복**: 5개 워크스페이스가 매일 커밋되는데 허브를 사람이 직접 갱신하면 금세 stale해진다. 자동화로 이 갭을 닫으려 한 것.
2. **구역 분리로 역할 명확화**: `knowledge/`, `insights/`, `study/`, `daily/`가 혼재하면서 "어디서 찾아야 하나"가 불명확했다. `learn`(공부 자료, 수동), `log`(진행 기록, 자동), `workspaces`(워크스페이스 정리)로 분리해 동선을 정비했다.
3. **cron → userland 전환**: 시스템 cron은 권한 이슈나 부재 시 복구가 어렵다. userland 방식 + 자가복구 supervisor로 안정성을 높였다.
4. **changelog 루프 완성**: 허브 자체 변경도 자동 기록(`changelog.sh`)함으로써 도구 변경 이력이 추적 가능해졌다. [이 변경 노트 자체가 해당 도구의 첫 출력물일 가능성이 있음]

## 알아둘 점·영향

**경로 변경으로 인한 브레이크 포인트**

- `knowledge/`, `insights/`, `study/`를 직접 참조하던 북마크·스크립트·외부 링크는 모두 깨진다. 새 경로는 `learn/knowledge/`, `learn/insights/`, `learn/curriculum.md`.
- `daily/` 아래 6월 1~8일 노트들이 삭제됐다. `log/daily/`로 이관됐는지, 아니면 완전 폐기됐는지 diff만으로는 확인 불가. 필요 시 git 히스토리(`git show f6c6bc4:daily/2026-06-01.md`)에서 복원 가능.

**자동화 전제 조건**

- 스케줄러는 프로세스 재시작 후 `ensure_scheduler.sh`가 호출돼야 복구된다. 서버 재부팅 시 수동 기동 필요 여부는 `tools/SCHEDULER_SETUP.md` 확인 필수.
- `synthesize.sh`는 커밋 메시지("LLM 종합")와 README 설명("LLM 파이프라인")으로 LLM 호출이 확인된다. 그러나 구체적으로 "API 키 또는 Claude Code 세션"을 요구하는지 여부는 스크립트 내부가 diff에 포함되지 않아 [근거부족] — `tools/synthesize.sh` 직접 확인 권장.
- `mirror_notes.sh`의 내부 동작 (note/ 디렉토리 존재 여부 처리, git 없는 워크스페이스 경로 분기 등)은 스크립트 내부가 diff에 포함되지 않아 [근거부족] — `tools/mirror_notes.sh` 직접 확인 권장.

**LLM 종합 결과물의 신뢰도**

README에 명시된 것처럼 `log/synthesis/` 파일은 "검토용 초안"이다. 생성→비평→종합 3단계를 거치지만, 원본 SCRATCHPAD/결과 파일 없이는 주장 근거를 독립 검증할 수 없다.
