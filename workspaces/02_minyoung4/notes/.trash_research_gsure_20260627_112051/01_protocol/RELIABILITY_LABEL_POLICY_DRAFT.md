# Reliability Label Generation Policy Draft

## Scope

This policy defines the first reliability/error label semantics for G-SURE after
valid full-volume OOF segmentation predictions exist.

This document does not approve official split creation, GPU inference,
prediction generation, or reliability label generation.

## Research Goal Reminder

G-SURE needs reliability labels that reflect actual segmentation failure under
held-out consortium shift. Labels must be derived from full-volume OOF
predictions, not from in-sample predictions, patch-only outputs, or
test-optimized thresholds.

## Eligibility Gate

Reliability/error labels may be generated only after all of the following pass:

1. Official LOCO split exists.
2. OOF prediction metadata validator passes.
3. OOF prediction artifact validator passes.
4. Prediction manifest rows are `split_role == test`.
5. Probability maps are full-volume assembled.
6. `mask_used_for_tile_placement == 0`.

## Primary Threshold Policy

For the first B1 segmentation baseline and first reliability-label generation:

```text
threshold_source = fixed_0.5
threshold_value  = 0.5
```

Rationale:

- It avoids held-out test threshold tuning.
- It is easy to reproduce across folds.
- It keeps the first reliability labels interpretable.
- It can be replaced later only by a train-fold validation threshold policy that
  is defined before seeing held-out test metrics.

Forbidden:

- selecting threshold by held-out test Dice,
- selecting threshold by held-out reliability metric,
- changing threshold after inspecting held-out error maps,
- using a different threshold per held-out subject unless a train-only rule is
  pre-registered.

## Primary Error Labels

Let:

```text
GT = selected_mask > 0
P  = OOF full-volume probability map, float in [0, 1]
B  = P >= 0.5
```

Primary maps:

```text
FN  = (GT == 1) and (B == 0)
FP  = (GT == 0) and (B == 1)
ERR = FN or FP
SOFT_ERROR = abs(GT.astype(float) - P)
```

Required dtypes:

```text
FN, FP, ERR: uint8 0/1
SOFT_ERROR: float32 in [0, 1]
```

Primary uses:

- `ERR` is the first binary reliability/error target.
- `FN` and `FP` must be reported separately for failure analysis.
- `SOFT_ERROR` is a threshold-aware but continuous severity map and should be
  kept for threshold-sensitivity diagnostics.

## Boundary Policy

Boundary labels are not primary in the first reliability-label generation.

Default for first B1 reliability labels:

```text
boundary_map_path = empty
boundary_radius = 0
```

Boundary labels may be introduced only after a separate decision locks:

- radius in voxels or millimeters,
- connectivity,
- dilation/erosion implementation,
- whether edema/core boundaries are treated differently,
- whether boundary maps are supervision, evaluation, or diagnostic only.

Rationale:

- Boundary bands can dominate voxel-level metrics because tumor boundaries are
  large relative to small errors.
- Boundary hyperparameters are easy to tune after seeing failures.
- The primary first claim should not depend on an arbitrary morphology choice.

## Subject-Level Failure Label

For subject-level reliability/failure detection, define:

```text
subject_failure_dice_le_0.8 = Dice(B, GT) <= 0.8
```

This threshold is a reporting convention, not a clinical safety threshold.
Per-consortium and lesion-size-stratified failure rates must be reported.

## Reliability Training Eligibility

Labels generated by this policy may be used for later G-SURE reliability-head
training only if their source predictions are OOF with respect to the subject
row being trained.

Allowed:

- train a second-stage reliability head on OOF error labels from training
  subjects,
- evaluate reliability maps on held-out consortium labels generated from held-out
  predictions,
- use `SOFT_ERROR` for diagnostic evaluation.

Forbidden:

- train on error labels from a segmentation model evaluated on its own training
  rows,
- use validation or held-out masks to place inference tiles,
- relabel after inspecting held-out performance.

## Required Label Manifest Values

For the first label generation:

```text
threshold_value = 0.5
boundary_radius = 0
boundary_map_path = empty
```

Required non-empty outputs:

```text
fn_map_path
fp_map_path
err_map_path
soft_error_map_path
```

## Minimum Label Validation

Before using generated labels:

- `FN`, `FP`, and `ERR` must be binary 0/1.
- `SOFT_ERROR` must be finite and in `[0, 1]`.
- All label maps must match target shape, affine, orientation, and spacing.
- `ERR` must equal `FN or FP`.
- `FN` and `FP` must be disjoint.
- Label manifest rows must link to validated OOF prediction rows through
  `source_prediction_id`.

## Open Decisions

- Whether later methods use train-fold validation thresholds instead of
  `fixed_0.5`.
- Whether boundary labels become a supervised target or remain diagnostic.
- Whether `SOFT_ERROR` becomes a training target or only a diagnostic map.
- Whether the first real label-generation output directory should live under
  `outputs/`, `experiments/`, or a separate approved artifact directory.

## Draft Implementation

The first-pass generator and validator are:

```text
research_gsure/02_audits/scripts/generate_reliability_labels.py
research_gsure/02_audits/scripts/validate_reliability_label_manifest.py
```

They have synthetic self-tests only. They must not be run on real predictions
until official split, OOF prediction validation, and artifact validation pass.
