## 1. 기존 노트북: "정렬돼 있었지만 안 돌아갔다"

헤드리스 실행으로 잡은 실제 버그들:

- **00**: `sex` 컬럼에 정수(AJU 0/1)와 문자(M/F, Male/Female)가 섞여 parquet 직렬화 실패. AJU 0=여/1=남은 **공식 설명서**로 확인 후 정규화.
- **03**: FastSurfer 경로에 `<subject_id>` 레벨 누락 + ROI 명명 오류(`ctx-lh-entorhinal`, `Thalamus`) + **VINN에 eTIV가 없음** → MaskVol을 ICV 프록시로.
- **04**: `eTIV` 참조 잔존.

교훈: 생성과 검증은 분리돼야 한다. 구조가 맞다고 돌아가는 게 아니다.

## 4. CDR Global을 공통 target으로 써도 되는가?

**왜 CDR인가**: diagnosis 라벨은 A4·KDRC에서 0%다. **CDR Global만 7개 컨소시엄 100% 존재** → 유일한 공통 라벨.

확인한 사실:
- CDR↔진단 극단 일치: CDR0→CN 99.0%, CDR≥1→AD 99.7%.
- 전수에서 CDR↔해마 단조감소(Spearman −0.33), ROI→(CDR0 vs ≥1) AUC ≈ 0.90.
- **그러나 분포가 사이트마다 근본적으로 다름**: AJU CDR0=2.3%(memory clinic), OASIS 79.8%(지역사회), A4는 preclinical.

→ `05_cdr_common_target.ipynb` 작성. 결론: **쓸 수 있다, 단 안전장치 필수(use_with_safeguards).**

### 적대적 비평(멀티에이전트 워크플로)이 잡은 것 — 내가 과신한 부분
1. **"CDR0.5 = 89% MCI"는 전역 통계가 아니다.** 이건 dx 보유 코호트 한정. 정작 CDR0.5의 다수인
   **KDRC(464)+A4(530)=994건은 dx가 100% 결측** → 이들에 89%를 적용하는 건 표본 외 추정. (직접 검증함)
2. **"부피"는 native mm³가 아니다.** option_b final_tensor-grid의 voxel_count + MaskVol 프록시.
   그리드 리샘플 자체가 부분적 크기 정규화를 이미 포함 → 해석에 그리드 아티팩트가 섞여 있다.
3. **AUC 0.9를 "유효 타깃"의 근거로 쓰지 말 것** — random split은 site 누수다.
   **leave-one-consortium-out**으로 site shortcut을 폭로하기 전엔 낙관치.
4. A4 CDR0.5=530은 의심스러웠지만, 출처가 실재(`A4/Clinical/Raw Data/cdr.csv`)임을 확인 → 조작 아님.

권장 타깃: **CDR 3-bin ordinal(0/0.5/≥1), A4 분리, 2차로 CDR0 vs CDR≥0.5, 보조로 CDR-SB 회귀.**

## 5. ComBat로 site 편향을 지울 수 있는가? — 실증

`06_harmonization_combat.ipynb`. batch=consortium, **CDR class를 보존 공변량**으로(over-correction 방지).

| 지표 | BEFORE | AFTER |
|---|---|---|
| site 분류 정확도 | 0.407 | **0.362** (다수클래스 기저선 0.364까지 ↓) |
| CN 내 site Kruskal p | 1.1e-13 | **0.18** (유의성 소멸) |
| CDR↔해마 Spearman (보존) | −0.297 | **−0.389** |
| ROI→(CDR0 vs ≥1) AUC (보존) | 0.905 | **0.908** |

→ **site 정보를 chance까지 지우면서 CDR 신호는 보존(오히려 강화).** 4절 안전장치의 실증.
한계: 누수 없는 평가는 train-fit→test-apply여야 하고, consortium은 거친 site 단위(스캐너 내부 미모델링).

## 6. 가장 뼈아픈 발견: 내가 써온 ROI는 "잠정(BLOCKED)" 데이터였다

voxel 설계를 워크플로에 맡겼더니, 한 에이전트가 manifest QC 플래그를 읽고 경고했다. 직접 확인:

```
do_not_use_for_atlaswide_roi_features = True    (전수 13,022)
roi_final_ready                       = False   (전수)
roi_final_grid_qc_status              = BLOCKED_PROVISIONAL  (전수)
roi_block_reason = "FastSurfer-to-native transfer requires ROI-specific
                    visual approval; ... candidates are provisional."
```

즉 03/05/06/컨소시엄 노트북이 의존한 option_b ROI는 **파이프라인이 "시각 승인 전 잠정"으로 막아둔 candidate**다.
결과가 틀렸다는 게 아니라, 모든 ROI 기반 결론을 **"검증됨"이 아니라 "후보(provisional)"로 강등**해야 한다.
정량 주장 전엔 `_reports/roi_transfer_option_b_*`의 per-ROI overlap/volerr/status로 **QC 게이트**를 통과시켜야 한다.

또 하나: summary JSON의 `centroid_voxel`은 256-conformed 좌표라 final_tensor 그리드와 ~31 voxel 어긋난다.
큐브 크롭 중심은 `expected_centroid_final_voxel`이나 마스크 재계산값을 써야 한다.
(다행히 내 `roi_tools.centroid()`는 마스크에서 재계산 → 기존 오버레이는 정상, `voxel_count`도 그리드와 일치.)
